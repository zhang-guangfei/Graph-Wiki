# 企业知识库四层模型对 Graph-Wiki 架构启发

> 来源：用户提供的知乎文章原文与配图，原链接为 `https://zhuanlan.zhihu.com/p/2043005659218703615`。  
> 说明：知乎网页在当前环境中直接访问受限，本文依据用户粘贴的完整原文与图片内容整理。  
> 定位：这是架构级参考资料，不是普通文章摘抄。目标是回答：Graph-Wiki 是否应向该四层模型靠拢，以及如何靠拢。

---

## 一、文章快速摘要

文章的核心判断是：企业知识库的价值不在于“存得更多”，而在于让知识在业务上下文中持续复利。

传统 RAG 的主要问题不是不能回答，而是“无状态”。每次问答都是独立检索和临时拼接，这次回答不会沉淀成下次可复用的知识结构。企业员工仍然需要自己回到原文核对，知识没有被编译、没有被维护，也没有变成组织资产。

文章提出企业知识库的四层演进金字塔：

| 层级 | 名称 | 核心能力 | 主要价值 | 主要局限 |
|------|------|----------|----------|----------|
| 第一层 | RAG | 切片、向量化、相似度检索、Prompt 拼接 | 低成本、开源成熟、快速验证 | 不懂关系、依赖切片质量、无状态 |
| 第二层 | LLM Wiki | 用 AI 将文档编译成结构化 Wiki 页面，建立双向链接 | 从“检索”升级为“知识编译”，降低查询成本，提高准确性 | 仍主要停留在文档和概念关联层 |
| 第三层 | GraphRAG / 知识图谱 / 本体 | 建实体、关系类型、因果链，支持多跳推理 | 知识进入业务上下文，能主动给建议 | 建模成本高，需要业务系统数据支撑 |
| 第四层 | GBrain 类工具 | Markdown + Git + 自动维护循环，持续去重、回链、合成、模式发现、孤立清理 | 形成可读、可审计、可版本回退的长期记忆系统 | 更像个人/高管外挂大脑，不一定适合普通企业全员使用 |

文章最重要的判断不是“四层技术栈”，而是下面三句话：

1. RAG 是技术起点，不是知识库终点。
2. LLM Wiki 的关键是 AI 编译，而不是 AI 检索。
3. 真正的企业知识库必须连接业务上下文，本体和系统数据比模型选择更重要。

## 二、对 Graph-Wiki 的直接启发

Graph-Wiki 当前的项目定位是“从源代码自动识别业务域边界，用 LLM 生成业务级 Wiki 文档”。这与文章第二层 LLM Wiki 高度同构，但 Graph-Wiki 的原料不是企业文档，而是源代码。

因此 Graph-Wiki 不应被定位成“代码 RAG 工具”，也不应只是“自动生成代码文档”。更准确的定位应是：

> Graph-Wiki 是面向代码资产的 LLM Wiki / GraphRAG 底座：先把代码编译成业务 Wiki，再逐步把业务域、API、字段、实体、调用链升级为可查询、可维护的代码本体网络。

这意味着产品北极星要从“跑通模块”提升为：

> 让一个陌生程序员或 Agent 能从业务大纲进入，逐层展开到业务域、业务点、接口、字段和实现细节，并且每次代码变化都能让知识基座持续更新。

## 三、文章四层与 Graph-Wiki 的对应关系

### 1. 第一层 RAG：Graph-Wiki 的“代码检索 / 原始结构层”

文章第一层是文档切片 + 向量检索。Graph-Wiki 对应的不是传统向量 RAG，而是代码结构提取层：

- `detect_corpus()`：识别代码文件
- `extract_ast()`：提取 AST / 符号
- `build_graph()`：构建代码图
- 设计中的 `filter_extraction_for_wiki()` 与 `build_light_graph()`：把原始代码结构压缩成 Wiki 可消费的轻量图

相同点：

- 都是底座层，负责把原始材料变成机器可检索结构。
- 都不直接解决“业务理解”问题。

关键差别：

- RAG 的基本单位是文本 chunk。
- Graph-Wiki 的基本单位应是代码结构：文件、类、方法、字段、API、调用边。
- Graph-Wiki 如果退化成只做源码向量检索，就会丢失它真正的差异化。

当前状态：

- 已实现：`detect_corpus()`、`extract_ast()`、`build_graph()`。
- 已设计未实现：`graph-lite.json`、预过滤链路、完整产物持久化。
- 风险：当前代码仍先构建完整图，再交给聚类过滤；这与“大项目先压缩、再理解”的架构方向不一致。

### 2. 第二层 LLM Wiki：Graph-Wiki 当前最核心的产品层

文章第二层强调“AI 编译”：摄入时就提前提炼概念、生成结构化页面、建立双向链接，查询时命中已编译好的知识。

Graph-Wiki 与这一层几乎正面重合：

- `business_cluster()` 将代码组织成业务域。
- `label_domains()` 用 LLM 生成业务域名称和摘要。
- `export_wiki()` 输出 Obsidian 兼容 Markdown。
- `summary.md`、`code-map.md`、`api-docs.md`、`data-flow.md` 共同构成结构化 Wiki。

相同点：

- 都不是临时问答，而是提前编译知识。
- 都强调人类可读页面和双向链接。
- 都希望查询时直接进入结构化知识，而不是重新读一堆碎片。

关键差别：

- 文章的 LLM Wiki 面向企业文档概念。
- Graph-Wiki 的 LLM Wiki 面向代码中的业务域和业务点。
- Graph-Wiki 的难点不是“概念抽取”，而是“从技术结构中反推业务语义”。

当前状态：

- 已实现：基础 Wiki 输出、域目录、代码地图、API 文档、数据流占位、域级 HTML 可视化。
- 部分实现：LLM 标注、业务点提取、API 映射。
- 未达标：当前小项目输出仍出现 `svn` 这样的技术/目录名，API 大量未分类，Wiki 还不能证明具备稳定业务导航价值。

### 3. 第三层 GraphRAG / 本体：Graph-Wiki 的下一阶段护城河

文章第三层强调实体关系、关系类型、因果链和多跳推理。对应到 Graph-Wiki，不能只理解为“把 NetworkX 图画出来”，而应理解为“代码业务本体”。

Graph-Wiki 的潜在本体对象包括：

- Domain：业务域
- BusinessPoint：业务点 / 用例入口
- API：前后端接口
- DTO / VO / Entity：数据对象
- Field：字段
- Table / Column：数据库表字段
- Service / Mapper / Feign / Event：实现和跨系统交互
- Rule / Spec / Decision：人工定稿的业务规则、规格和架构决策

Graph-Wiki 的潜在关系类型包括：

- `contains`：业务域包含业务点、类、API
- `calls`：方法调用
- `exposes`：Controller 暴露 API
- `consumes`：前端页面调用 API
- `maps_to`：DTO 字段映射到 Entity / DB 字段
- `depends_on`：业务域依赖
- `implements_rule`：代码实现某条业务规则
- `changes_with`：代码变更影响某个业务域或规则

相同点：

- 都要求从“相关”升级到“以什么关系相关”。
- 都要求支持多跳追踪和业务上下文解释。

关键差别：

- 企业知识图谱常依赖组织、财务、项目、合同等业务系统天然数据。
- Graph-Wiki 当前主要从源代码推断，业务上下文是间接的、缺失的。
- 因此 Graph-Wiki 如果要进入第三层，必须允许接入外部业务上下文，而不能只依赖 AST。

当前状态：

- 已实现：有限代码图、域间依赖、API 映射、字段映射的初版。
- 已设计：跨域调用分析、field-map、api-map、domains.json、graph-lite。
- 缺失：统一本体 schema、 typed relationship、实体级关系、业务规则与代码的连接、多跳查询接口、外部业务上下文接入。

### 4. 第四层 GBrain 类工具：Graph-Wiki 的长期记忆与治理层

文章第四层的重点不是“个人工具”，而是“知识库必须被 AI 持续维护”。GBrain 的启发在于 Markdown + Git + 自动维护循环：

- 可读
- 可审计
- 可回退
- 可自动去重
- 可自动补链
- 可自动合成
- 可发现模式
- 可清理孤立页面

Graph-Wiki 天然适合吸收这一层思想，因为它已经选择 Markdown Wiki 和人工定稿机制。

对应能力：

- `wiki/` 作为 Git 可管理知识库
- `rules.md` / `spec.md` 人工定稿，不被 LLM 随意覆盖
- `manifest.json` 记录文件与域关系
- `update` / `recluster` 支持增量更新
- 未来可以加入类似 Dream Cycle 的维护任务

当前状态：

- 已实现：Markdown 输出、占位文件保护、简化 update。
- 已设计：manifest、增量更新、变更日志。
- 缺失：定期维护循环、孤立页面检测、回链补全、重复域合并、陈旧知识检测、知识质量评分、知识演化日志。

## 四、当前 Graph-Wiki 与四层模型的成熟度评估

| 四层模型 | Graph-Wiki 对应层 | 当前成熟度 | 判断 |
|----------|-------------------|:--:|------|
| RAG / 检索底座 | graphify 复用层、AST、graph-lite | 45% | 原始提取可跑，轻量图和大项目策略还未落地 |
| LLM Wiki | 业务域 Wiki、摘要、代码地图、API 文档 | 55% | 产品方向最匹配，但输出质量尚未达业务可用 |
| GraphRAG / 本体 | domains/api/field/calls/实体关系 | 20% | 有雏形，没有统一本体和多跳推理能力 |
| GBrain / 自动维护 | Markdown + Git + update + manifest | 15% | 方向天然契合，但维护循环基本未实现 |

结论：

> Graph-Wiki 目前不是四层都要做的成熟平台，而是一个正在从第一层跨向第二层的代码知识编译器，并具备向第三层和第四层演进的天然架构基础。

## 五、CEO 视角的架构判断

### 判断 1：Graph-Wiki 的战略重心应先锁定第二层，而不是急着做完整第三层

文章说 LLM Wiki 是比 RAG 更优的企业知识库方案。Graph-Wiki 的最大差异化也在这里：从代码生成业务 Wiki。

因此短期不要把团队带进“知识图谱大而全”的坑。第三层很诱人，但如果第二层 Wiki 质量不稳定，图谱会变成更复杂的噪音容器。

阶段目标应调整为：

1. 先证明“代码 → 业务 Wiki”能稳定产出业务可读结果。
2. 再把 Wiki 背后的关系沉淀为本体。
3. 最后做自动维护和多跳推理。

### 判断 2：Graph-Wiki 不能做普通代码 RAG，这是战略红线

如果 Graph-Wiki 只是“把源码切片后向量检索”，它会进入高度拥挤的市场，且没有护城河。

Graph-Wiki 的差异化必须是：

- 类/文件/方法/字段级结构理解
- 业务域自动划分
- 业务点编译
- API 和字段链路
- Wiki 可读性
- Agent scope reduction

所以第一层只能作为底座，不能成为产品叙事主体。

### 判断 3：第三层本体必须以“代码业务本体”起步，而不是直接做企业全域本体

文章里的第三层依赖组织、财务、项目、供应链等系统打通。Graph-Wiki 当前没有这些数据，不能假装自己已经具备企业一体化平台能力。

更务实的路线是先定义代码业务本体：

```
Domain -> BusinessPoint -> API -> DTO -> Entity -> Field -> Table/Column
       -> ServiceCall -> ExternalSystem
       -> Rule/Spec/Decision
```

等这个本体稳定后，再开放外部上下文接入：

- 业务词典
- 系统边界配置
- 数据库 schema
- 接口网关元数据
- 需求/PRD/测试用例
- 人工维护的 rules/spec/decision

### 判断 4：第四层自动维护是 Graph-Wiki 的长期复利，不是装饰功能

文章最有价值的观点是“这次查询要沉淀成下次资产”。Graph-Wiki 也必须避免“每次 build 都重新生成一堆静态文档，用完就忘”。

因此 `update` 不应只是增量跑 pipeline，而应升级成知识维护循环：

1. Detect：发现代码和 Wiki 的变化
2. Reconcile：对齐旧域、新域、重命名域
3. Backlink：补全双向链接和孤立业务点
4. Synthesize：把重复业务点/相似域合成更高层摘要
5. Validate：输出质量评分和失败原因
6. Preserve：保护人工定稿内容
7. Changelog：记录知识演化

这才是 Graph-Wiki 从“文档生成器”变成“代码知识操作系统”的关键。

## 六、建议的架构路线

### 路线总原则

不要把文章四层机械照搬成四个模块。应把它转成 Graph-Wiki 的四阶段产品路线：

| 阶段 | Graph-Wiki 目标 | 对应文章层级 |
|------|-----------------|--------------|
| v1.0 | 代码结构底座 + 可用 LLM Wiki | 第一层 + 第二层 |
| v1.5 | 代码业务本体 schema + typed relationships | 第三层起步 |
| v2.0 | 多跳查询、影响分析、规则/规格与代码连接 | 第三层成型 |
| v2.5 | Dream Cycle 式自动维护、知识复利、长期记忆 | 第四层 |

### 近期必须做的 5 件事

1. **先落地 graph-lite**
   - 这是第一层到底能不能支撑第二层的前提。
   - 没有轻量图，大项目会被原始代码噪音拖垮。

2. **把 Wiki 质量评估做成硬门槛**
   - 第二层不是“生成了 Markdown”就算完成。
   - 必须用 `build-report.json` 判断业务域名称、API 分类、业务点数量、技术域名比例。

3. **定义 Code Ontology v0**
   - 不要等所有图谱能力都实现再定义本体。
   - 先在文档和 JSON schema 中固定 Domain、BusinessPoint、API、DTO、Entity、Field、Rule、Decision 的关系。

4. **把 `rules.md` / `spec.md` 提升为一等公民**
   - 文章第三层强调业务上下文。
   - 对 Graph-Wiki 来说，人工定稿的规则和规格就是最早、最可信的业务上下文。

5. **把 update 重新定义为 knowledge maintenance**
   - 不只是重跑生成。
   - 要负责保留、对齐、合成、补链、评分和变更记录。

---

## 七、不建议做的事

1. **不建议现在做完整企业知识图谱**
   - 没有企业系统数据接入，做出来多半是代码关系图，不是真本体。

2. **不建议优先做聊天问答界面**
   - 聊天会诱导项目退回 RAG 形态。
   - 先让 Wiki 和本体稳定，再做问答入口。

3. **不建议把所有关系都交给 LLM 推断**
   - 代码结构关系应尽量规则化、可审计。
   - LLM 负责命名、摘要、归纳和歧义消解。

4. **不建议继续横向堆模块**
   - 当前真正缺的不是模块数量，而是层间契约和质量闭环。

---

## 八、对当前迭代的影响

当前 Step 3 实现建议保持不变，但解释上升级：

原说法：

> 实现 graph-lite、产物契约、API 域推断、质量报告。

升级后的架构含义：

> 完成 Graph-Wiki 从第一层代码结构底座到第二层 LLM Wiki 的最小闭环，并为第三层代码业务本体保留 schema 和关系出口。

这意味着 Step 3 的实现优先级应为：

1. `filter_extraction_for_wiki()` + `build_light_graph()`：第一层底座可控。
2. `domains.json` / `api-map.json` / `field-map.json` / `build-report.json`：第二层 Wiki 可审计。
3. `infer_frontend_api_domain()`：让 API 从孤立接口进入业务域上下文。
4. `evaluate_wiki_quality()`：让“是否可用”变成机器可判断。
5. 预留 `ontology` / `relationships` 字段：不实现完整图谱，但不把未来路堵死。

---

## 九、最终建议

Graph-Wiki 应向这篇文章的核心思想靠拢，但不要照搬企业知识库四层产品形态。

最合理的战略定位是：

> 先成为代码世界的 LLM Wiki，再演进为代码业务本体，最后成为可持续维护的代码知识记忆系统。

一句话路线：

> 不做“源码搜索引擎”，不做“静态文档生成器”，而做“会把代码编译成业务知识、并持续维护这份知识的系统”。
