## 🔴 严重问题（4 项，必须修复）

### S1. field_mapper.py 缺失 — 需求 2 没有对应模块

**三份报告一致认定，严重度 95/100。**

需求 2（"表/字段来源追溯 + 前后端数据流"）在第十章明确标注为"需新增 `field_mapper.py`"，但：

- 第四章模块设计中没有 field_mapper 的任何设计段落
- 第十五章项目结构中没有 `field_mapper.py`
- 数据流图中 `field-map.json` 凭空出现，没有生成者
- `field-map.json` 被四处引用但 **schema 从未定义**

**修复**：在第四章新增 4.6bis 节，或在 `api_mapper` 中扩展职责（两者解析逻辑天然相关）。

### S2. graphify query 子图作用域机制不明

**Agent 2 和 3 独立发现，严重度 90/100。**

文档核心卖点是"Agent 搜索范围从 34K 节点缩小到 ~500 节点"。但 graphify query 的当前接口是否支持"指定文件列表/节点子集"作为搜索参数？文档没有说明。

- 如果 graphify query 总是全图 BFS → 图遍历复杂度没有降低，只是 Agent 在认知层面过滤
- Chapter 7.4 和 11.3 的所有 Token 节省计算都依赖这个假设
- Chapter 11.4 的深度查询 ~15s 耗时也基于 34K 节点全图

**修复**：明确说明实现方式：(A) 修改 graphify.query 支持节点ID白名单参数；(B) 或承认缩小的是结果过滤空间，重新计算收益。

### S3. 章节编号严重错乱

**Agent 3 发现，严重度 85/100。**

正文中小节编号与章节号不匹配：
- `### 10.1` 应为 `12.1`（粒度设计）
- `### 10.2` 应为 `12.2`
- `### 11.1/11.2/11.3` 应为 `13.1/13.2/13.3`（差距分析）
- Chapter 5 标题标注为"（旧）"，与 Chapter 8 并存

直接违反文档自定位的"唯一设计真相源"。

### S4. 聚类算法 Java 偏见 + 非 Java 项目无适配策略

**Agent 1 和 2 独立发现，严重度 82/100。**

`extract_domain_key()` 硬编码对 Java 反向域名包路径的解析（`com.smc.smccloud.bin.*` → `bin`）。对非 Java 项目：
- JavaScript/TypeScript 项目没有"包路径"
- Python 项目使用扁平 import
- Go 模块使用域名路径

Phase 4 承诺"前端项目适配"但没有任何前端聚类设计细节。

**修复**：为 JS/Vue 项目定义基于目录前缀的等价聚类规则；为 Python/Go 定义基于 import 前缀的规则。
