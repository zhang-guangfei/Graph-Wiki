## 一、修复清单

### 🟡 M1：聚类算法步骤不一致（4.2 vs 8.2）

**现状**：4.2 描述"三步法"（过滤 → 包聚类 → import 调整），但代码实际有 5 步（过滤 → 锚点提取 → 包聚类 → import 调整 → 业务点提取），8.2 数据流也展示了 5 步。

**修复**：4.2 节重写为"五步法"，加入 Step 2（锚点提取）和 Step 5（业务点提取），与代码和 8.2 数据流对齐。

**变更文件**：`graph-wiki-design.md` §4.2 L312-348

---

### 🟡 M2：docs.py 引用清理

**现状**：文档中 3 处引用不存在的 `docs.py` 模块（L117、L1761、L2130）。实际功能分布在 `api_mapper.py`（API 文档）、`field_mapper.py`（数据流）、`export.py`（Wiki 生成）。

**修复**：
- L117：`graph_wiki.docs()` → `graph_wiki.export()` + `graph_wiki.api_mapper()` + `graph_wiki.field_mapper()`
- L1761：`docs.py (可选)` → `export.py`（已涵盖 API 文档/数据流/ER 图导出）
- L2130：`⚠️ 需新增 docs.py 模块` → `✅ 已通过 api_mapper + field_mapper + export 联合覆盖`

**变更文件**：`graph-wiki-design.md` L117、L1761、L2130

---

### 🟡 M3：LLM 并行化策略说明缺失

**现状**：文档提到"~30s 标注时间"但未说明并发策略。代码使用 `ThreadPoolExecutor(max_workers=5)` 并发标注 25 个域。

**修复**：在 4.3 节新增"并行化策略"段落，说明：
- `ThreadPoolExecutor` 5 路并发
- 每域独立调用，互不影响
- 失败时独立重试
- 实际耗时估算：~30s（5 并发 × 5 轮 / 25 域）

**变更文件**：`graph-wiki-design.md` §4.3

---

### 🟡 M4：api-map/field-map 增量更新策略缺失

**现状**：4.6 和 4.7 未说明增量更新。但 pipeline-design.md 中有简要描述。

**修复**：
- 4.6 新增"增量更新"小节：前端 api/ 文件变更 → 重解析该模块；后端 Controller 变更 → 重解析该文件
- 4.7 新增"增量更新"小节：Entity 文件变更 → 重新解析该 Entity；DTO 变更 → 重跑 match_dto_to_entity
- 引用 manifest.json 作为变更检测机制

**变更文件**：`graph-wiki-design.md` §4.6、§4.7

---

### 🟡 M5：关键 JSON Schema 未定义

**现状**：`field-map.json` 和 `manifest.json` 被引用但 schema 从未定义。

**修复**：
- 6.3 新增 `field-map.json` 规范（domain → table → column 三层结构，已在 field-mapper-design.md 中有示例）
- 6.4 新增 `manifest.json` 规范（`{files: {path: {hash, last_modified}}, domains: {id: {version, anchors_hash}}}`）
- 更新 6.1 文件清单表

**变更文件**：`graph-wiki-design.md` §6

---

### 🟡 M6：Phase 时间线调整

**现状**：原估算 5-7 周，审核建议 9-10 周。

**修复**：更新 §9 各 Phase 时间估算，反映审核反馈（Phase 1:2周, Phase 2:2周, Phase 3:1.5周, Phase 4:2周, Phase 5:1.5周, 总计 9-10 周）。

**变更文件**：`graph-wiki-design.md` §9

---

### 🟡 M7：Agent Token 分析缺少场景

**现状**：7.4 节只分析了"单域查询"场景（~1,500 tokens），未包含首次导航和跨域查询。

**修复**：在 7.4 新增两个场景：
- **首次导航开销**：Agent 首次接触项目时需读 `index.md`（~500 tokens）+ 扫描 3-5 个域的 `summary.md`（~2,000 tokens）= ~2,500 tokens 建立全局认知
- **跨域查询场景**：查询涉及 3 个域的数据流时，需读取 3 个域的 `data-flow.md`（~600×3=1,800 tokens）+ `api-docs.md`（~800 tokens）= ~2,600 tokens

**变更文件**：`graph-wiki-design.md` §7.4

---

### 🟡 M8：测试策略 / 错误处理 / 安全

**现状**：文档完全没有涉及这三个方面。

**修复**：新增 §17 "质量保障"，包含 3 个小节：
- **17.1 测试策略**：单元测试（各模块文件内）+ 集成测试（tests/ 目录，对真实项目 `graph-wiki build`）+ 回归测试（对比聚类结果稳定性）
- **17.2 错误处理**：PipelineError 层级、各模块错误类型、safe_run 装饰器
- **17.3 安全考量**：LLM API Key 管理（环境变量）、源码采样不包含密钥文件、Wiki 输出中的敏感信息过滤

**变更文件**：`graph-wiki-design.md` 新增 §17
