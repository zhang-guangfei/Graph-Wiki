## 一、文档边界与验收方式

### 1.1 产品方审核范围

产品方只需要验证 Graph-Wiki 是否达到本文定义的产品设计标准：

- 是否能围绕业务域提供“流程 → 规则 → 证据”的深度阅读体验。
- 是否能自动生成 business flows、business rules、evidence chain、field rules 的可审查初稿。
- 是否能让用户从业务目标出发，而不是从文件目录、类名或图谱节点出发理解项目。
- 是否能明确区分“构建成功”和“产品可读”。
- 是否能在 Workbench / Wiki / Agent 查询等消费形态中保持一致的产品语义。

### 1.2 产品方不审核范围

以下内容不属于产品方审核范围，由 AI/OMX 团队负责：

- 模块拆分、类/函数设计、代码组织和内部接口。
- AST、聚类、API 映射、字段映射、证据归一化、LLM 标注等实现策略。
- JSON 文件的完整字段定义、schema 细节和兼容处理。
- CI、release gate、测试夹具、工程性能、依赖治理和生成物治理。
- 与 Graphify 的底层复用方式和技术取舍。

这些内容统一维护在 [`graph-wiki 工程架构设计.md`](./graph-wiki%20工程架构设计.md)。

### 1.3 验收方式

产品验收不以“代码是否存在某模块”为准，而以最终产品是否满足以下标准为准：

1. 用户能打开一个业务域，并按“流程 → 规则 → 证据”理解该业务域。
2. 每个核心流程、规则、字段规则都能看到可追溯证据或明确的缺失/partial 状态。
3. 产品能显示阅读质量状态，而不是只显示构建成功。
4. 对企业全栈项目，主验收样本应达到产品可读；复杂真实样本允许 warning，但必须暴露原因。
5. 产品输出能够被人类、Agent 和 CI 以一致语义消费。

## 二、产品定位与目标

### 2.1 Graph-Wiki 是什么

Graph-Wiki 是一个面向代码库的业务知识产品。它从源代码中自动识别业务域边界，生成业务流程、业务规则、字段规则和证据链，让用户能够从业务视角理解系统。

它不是普通代码搜索、不是单纯图谱展示，也不是只给 Agent 使用的代码索引。Graph-Wiki 的第一客户是需要理解系统的人，Agent 是第二客户。

### 2.2 核心目标

> 让程序员、架构师、产品/业务分析人员和 Agent 能通过业务域理解一个代码库，而不是被迫从文件目录、类名、方法调用或全量图谱开始阅读。

核心目标包括：

- 自动生成业务域阅读入口。
- 自动生成业务流程、业务规则、字段规则和证据链。
- 支持复杂企业全栈项目的业务理解。
- 为重构、缺陷分析、需求评审和知识沉淀提供可追溯依据。

### 2.3 产品定位

Graph-Wiki 的产品定位是：

> **业务域深度阅读工作台。**

它帮助用户回答：

- 这个系统有哪些业务域？
- 每个业务域解决什么业务问题？
- 核心流程如何运转？
- 关键业务规则在哪里实现？
- 字段从哪里来、如何转换、最终落到哪里？
- 哪些结论有代码证据，哪些只是机器草稿或不完整推断？

### 2.4 与 Graphify 的产品关系

Graphify 偏底层代码图谱和 Agent 查询，Graph-Wiki 偏人类可阅读的业务知识产品。Graph-Wiki 可以复用 Graphify 的底层分析能力，但产品体验不应退化为“展示 graph.json”。

## 三、目标用户与核心场景

### 3.1 目标用户

- 新加入项目的研发人员：快速理解业务域和代码入口。
- 架构师/技术负责人：评估模块边界、业务耦合和重构风险。
- 产品经理/业务分析人员：理解系统当前真实业务流程和规则。
- 测试与质量人员：追踪业务规则、字段规则和异常分支。
- AI Agent：基于稳定证据链进行代码问答、变更分析和重构规划。

### 3.2 核心场景

1. **业务域阅读**：用户进入某个业务域，先看业务目标，再看流程、规则和证据。
2. **字段追溯**：用户查看某字段从前端、API、DTO、Service、Entity 到 DB 的链路。
3. **规则审查**：用户确认某条规则来自哪里、可信度如何、是否缺失证据。
4. **跨模块理解**：用户理解一个业务流程横跨哪些模块、API 和数据对象。
5. **变更/重构准备**：用户基于业务域和证据链判断修改影响范围。
6. **Agent 辅助分析**：Agent 基于结构化业务知识进行更准确的代码问答。

## 四、产品需求与验收目标

### 4.1 业务域深度阅读

产品必须让用户以业务域为入口，而不是以文件目录或代码图谱为入口。业务域页面应优先展示业务目标、核心流程、关键规则、字段规则和证据。

### 4.2 流程 → 规则 → 证据

核心阅读路径必须是：

```text
业务域 → 业务流程 → 业务规则 / 字段规则 → 证据
```

每个流程步骤应能关联规则和证据；每条规则应能说明它来自哪段代码、API、字段、配置或文档。

### 4.3 自动生成初稿

Graph-Wiki v1 应优先自动生成业务流程、业务规则、字段规则和证据链初稿。允许机器草稿、partial 和 warning，但必须明确标识，不允许把无证据推断伪装成确定结论。

### 4.4 字段规则与数据链路

产品必须支持字段级阅读：字段从哪里输入、在哪些层转换、如何校验、最终影响什么业务对象。字段链路不完整时，要显示缺失层级和 partial 原因。

### 4.5 跨模块关联分析

产品应支持用户理解一个业务流程涉及哪些模块、API、服务、实体和数据对象。跨模块关系必须服务于业务阅读，而不是单纯展示调用图。

### 4.6 代码缺陷识别与重构设计

产品可基于业务域、规则、字段链路和证据提供缺陷识别与重构建议。但这些建议必须可追溯、可审查，并区分确定结论与机器建议。

### 4.7 输出形态

产品应至少支持三类消费形态：

- Workbench：面向人类的业务域深度阅读界面。
- Wiki / Markdown：面向长期知识沉淀和 Obsidian 类工具。
- Agent / CI：面向自动化查询、质量门禁和持续更新。

### 4.8 v1 验收标准

v1 验收重点：

- 企业全栈样本能生成可读的 Domain Read Model。
- Workbench 能以业务域展示“流程 → 规则 → 证据”。
- business flows、business rules、field rules、evidence chain 均存在并可追溯。
- `build.status` 与 `productQuality` 清晰分离。
- 真实复杂样本可以 warning，但 warning 原因必须可见。

## 五、核心产品模型：Domain Read Model

### 5.1 产品真相源

Domain Read Model 是 Graph-Wiki v1 的产品真相源。Workbench、产品质量判断和 Agent 产品语义都应围绕它组织。

工程上它对应 `domain-read-model.json`，但产品方只需要理解其产品含义：它是业务域深度阅读所需信息的统一模型。

### 5.2 Domain / 业务域

Domain 表示一个可被用户理解和讨论的业务区域。它不是简单文件夹，也不是纯算法社区。一个合格的 Domain 应回答：

- 这个业务域做什么？
- 有哪些入口和核心对象？
- 有哪些流程、规则和证据？
- 当前阅读质量如何？

### 5.3 Business Flow / 业务流程

Business Flow 描述业务如何发生。流程不应只是自然语言摘要，而应能拆成步骤，并关联规则和证据。

### 5.4 Business Rule / 业务规则

Business Rule 描述业务约束、条件、状态流转、校验逻辑和异常分支。规则必须表达可信度、状态和证据来源。

### 5.5 Field Rule / 字段规则

Field Rule 描述字段含义、取值、转换、校验和跨层流动。字段规则是业务深度阅读的重要组成部分，尤其适用于企业全栈系统。

### 5.6 Evidence Chain / 证据链

Evidence Chain 是产品可信度基础。任何关键结论都应能回到具体证据；无法证明时必须显示为 partial、warning 或 machine draft。

### 5.7 productQuality / 产品质量状态

`productQuality` 表示 Domain Read Model 的产品可读性。它不同于构建状态：构建成功不等于产品可读。

产品验收必须关注：

- `deepReadingStatus`
- `coreDomainEvidenceStatus`
- `ruleCorrectnessRisk`
- warnings / errors 是否可解释

## 六、产品体验架构

### 6.1 Workbench 主路径

Workbench 是 v1 的主要产品体验。用户进入后应能：

1. 查看项目业务域列表。
2. 选择一个业务域。
3. 阅读业务目标和质量状态。
4. 按流程查看关键步骤。
5. 从流程进入业务规则和字段规则。
6. 查看每条结论的证据。

### 6.2 业务域详情页

业务域详情页应避免以“代码文件列表”为主体，而应以业务阅读为主体：

```text
业务目标
→ 阅读质量
→ 核心流程
→ 关键业务规则
→ 字段规则
→ 证据与代码入口
```

### 6.3 质量提示体验

当系统只能生成 partial 或 warning 时，产品必须明确告诉用户：

- 哪个业务域不完整。
- 缺少哪类证据。
- 哪些字段链路不完整。
- 哪些规则只是机器草稿。
- 用户下一步可以如何审查。

### 6.4 Agent 消费体验

Agent 不应直接从零扫描全量代码回答业务问题，而应优先消费 Domain Read Model 和证据链，再按需下钻到代码。

## 七、产品输出与消费形态

### 7.1 Workbench

Workbench 是产品主界面，负责把业务域、流程、规则、字段规则和证据链组织成可阅读体验。

### 7.2 Wiki / Markdown

Wiki / Markdown 用于长期沉淀、离线阅读和 Obsidian 类工具。它应与 Domain Read Model 保持语义一致，不应成为另一套互相矛盾的业务解释。

### 7.3 结构化数据

结构化数据用于 Workbench、Agent 和 CI。产品方不审核具体字段实现，但要求数据能稳定表达业务域深度阅读所需语义。

### 7.4 CI / 自动化消费

CI 不只判断构建是否成功，还要判断产品输出是否达到最低可读标准。

## 八、产品质量与发布验收标准

### 8.1 产品可读性标准

发布前必须证明：

- 业务域能被识别和展示。
- 核心业务域具备流程、规则、字段规则和证据。
- 用户能从业务域一路追溯到证据。
- 产品能清楚展示 passed / warning / failed。

### 8.2 构建成功不等于产品成功

`build.status=passed` 只能说明流水线执行成功。产品发布必须同时检查 `productQuality`。如果缺少产品质量信号，不能认为产品已达标。

### 8.3 fullstack-enterprise 验收

企业全栈样本是主验收路径，应该达到：

- 构建成功。
- Domain Read Model 可读。
- Workbench 可展示业务域深度阅读路径。
- productQuality 达到 passed。

### 8.4 svn-platform smoke 验收

真实复杂样本可以存在 warning，但必须满足：

- 构建成功。
- warning 原因明确。
- 产品不会把 warning 误报为 passed。
- 用户能看到缺失的是字段规则、证据链还是其他产品能力。

### 8.5 发布门禁

发布门禁必须覆盖：

- 自动化测试通过。
- Workbench 可构建。
- fullstack-enterprise 达到产品可读。
- svn-platform smoke 可运行并暴露质量状态。
- 生成物和依赖污染不进入版本控制。
- 独立代码审查和架构不变量审查通过。

## 九、产品边界与非目标

### 9.1 不是普通代码 RAG

Graph-Wiki 不应退化为“把代码切片后做检索问答”。检索可以作为能力，但产品主线必须是结构化业务域深度阅读。

### 9.2 不是纯图谱可视化工具

图谱是底层支撑，不是最终产品目标。用户需要的是业务理解，而不是被迫阅读复杂节点和边。

### 9.3 不是完全人工维护的 Wiki

Graph-Wiki 应自动生成初稿，并通过证据链支持审查。人工可以修正和确认，但不能依赖人工从零编写。

### 9.4 不以模块实现作为产品验收标准

产品验收不检查是否存在某个内部模块，而检查最终体验和产品质量是否达标。

## 十、产品路线图

### 10.1 v1：Domain Read Model 主链路

完成业务域深度阅读最小闭环：业务域、流程、规则、字段规则、证据链、Workbench 展示和质量门禁。

### 10.2 v2：规则准确率与字段链路增强

增强复杂业务规则、状态机、SQL/MyBatis/JPA、前后端字段链路和异常分支识别。

### 10.3 v3：Workbench 产品化

完善搜索、筛选、证据展开、质量提示、审查工作流、跨域跳转和交互体验。

### 10.4 v4：多语言与大型项目适配

扩展到更多语言、框架和大型企业项目，并建立更稳定的性能与质量基线。

### 10.5 v5：持续更新与知识治理

支持持续变更、增量更新、业务知识版本化、人工确认和长期知识治理。

## 十一、产品设计决策

### 11.1 以 Domain Read Model 为产品真相源

因为它能同时服务 Workbench、人类阅读、Agent 查询和质量门禁，避免多个输出各自解释业务。

### 11.2 以“流程 → 规则 → 证据”为主阅读路径

这是用户理解业务系统的自然路径，比文件目录、调用图或代码搜索更接近真实工作方式。

### 11.3 区分 build.status 与 productQuality

构建成功是工程信号，产品可读是用户价值信号。两者必须分离。

### 11.4 Workbench 优先于传统 Wiki

Wiki 适合沉淀，但 Workbench 更适合交互式深度阅读、证据展开和质量提示。

### 11.5 允许机器草稿，但必须可审查

自动生成是产品效率来源，但所有不确定内容都必须标识状态和证据。
